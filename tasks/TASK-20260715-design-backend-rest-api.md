## Что делаем
Бэкенд-кода ещё нет — есть только `docs/architecture.md`, ADR и docker-compose. Нужно спроектировать контракт REST API Go-бэкенда до реализации: эндпоинты витрины (аутентификация по `initData`) и админки (admin JWT), схемы запросов/ответов, коды ошибок, модель аутентификации и RBAC. Результат — единая OpenAPI-спека, которая станет контрактом для последующей реализации доменных модулей.

## Пользователь получает
Как **разработчик бэкенда/фронтенда**, я могу **опираться на согласованную OpenAPI-спеку**, чтобы **реализовывать и интегрировать витрину, админку и API параллельно, не расходясь по контракту и не нарушая архитектурных инвариантов**.

## Готово когда
- [ ] Given L3-модули из `docs/architecture.md` (auth, content, catalog, cart, orders, delivery, payment, media, admin), when проектируется спека, then каждый модуль покрыт эндпоинтами с методами, путями, параметрами и телами запросов/ответов
- [ ] Given покупатель витрины, when он вызывает любой эндпоинт витрины, then контракт требует Telegram `initData` (проверка только на сервере) и не предполагает клиентской проверки подписи
- [ ] Given персонал админки, when он вызывает эндпоинт админки, then контракт требует admin JWT и описывает RBAC-роли (admin / order_manager / content_manager) — механизм аутентификации отличается от витрины
- [ ] Given загрузка изображения товара/новости, when клиент грузит файл, then контракт описывает выдачу presigned URL модулем media и загрузку напрямую в MinIO, а не проксирование байтов через API
- [ ] Given оформление заказа, when создаётся заказ, then схемы `order_items` фиксируют снимок позиции (название, цена, характеристики), а не ссылку на актуальный каталог
- [ ] Given единый каталог ошибок, when любой эндпоинт возвращает ошибку, then используется единая схема ошибки с кодами и HTTP-статусами (400/401/403/404/409/422/429/5xx)
- [ ] Given файл спеки, when он валидируется линтером OpenAPI, then спека проходит валидацию без ошибок

## Как делаем
- Взять за основу L3-диаграмму «Components of Backend API» и сгруппировать пути по доменным модулям; один модуль = один tag OpenAPI.
- Описать две схемы безопасности `securitySchemes`: `initData` для витрины и `adminJWT` (Bearer) для админки; повесить их пер-операционно, отразить RBAC в описаниях.
- Спроектировать общие компоненты: схема ошибки, пагинация/фильтры каталога (фасеты), денежные и timestamptz-поля, схемы товара с `attributes` (id значений справочников, не текст).
- Отразить инварианты в контракте: presigned-загрузка через media, снимки в заказах, оформление заказа как одна операция; никаких эндпоинтов, публикующих напрямую в RabbitMQ (outbox — внутренняя деталь, наружу не торчит).
- Согласовать пути с маршрутизацией proxy: витрина и API — один хост, `/api`; админка ходит в те же доменные сервисы через `api`.
- По завершении спеки нарезать дочерние задачи на реализацию модулей.

## Вопросы
- ✅ Расположение/формат/версия: `docs/api/openapi.yaml`, OpenAPI **3.1.0** (YAML).
- ✅ admin JWT: `POST /api/v1/admin/auth/login` + `/refresh` + `/logout` + `GET /api/v1/admin/auth/me`; access Bearer JWT + refresh. Конкретные TTL — на этапе реализации auth-модуля.
- ✅ `initData`: заголовок `Authorization: tma <initData>`, проверка только на сервере.
- ✅ Платёжный вебхук: публичный `POST /api/v1/payments/webhook/{provider}`, security-схема `providerSignature` (заголовок `X-Payment-Signature`), идемпотентность по `provider_payment_id`.
- ✅ Версионирование: префикс `/api/v1`.
- ❓ Остаются на подтверждение владельцем `docs/database.md` (все помечены inline в спеке): денежный формат — строка-decimal, single-currency (валюты в схеме БД нет); id как `integer/int64` (по ERD); дополнительные поля `AdminUser.full_name`, `Category.description`/`image`, `Product.description`, `Order.comment`; ресурс `Settings` без таблицы в модели данных.
