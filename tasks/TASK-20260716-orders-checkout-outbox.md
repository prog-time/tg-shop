# TASK-20260716-orders-checkout-outbox: Оформление заказа + outbox-релей + notifications

**Type:** feature  
**Size:** L  
**Status:** todo
**GitHub Issue:** #8 — https://github.com/prog-time/tg-shop/issues/8

## Что делаем
Реализовать ядро покупки: модуль `orders` (оформление одной транзакцией — сбор состава из корзины, списание остатка с блокировкой строки, расчёт доставки, запись заказа со снимками и события в `outbox`), релей `outbox` в `worker` (публикация в RabbitMQ после коммита) и потребитель `notifications` (идемпотентная отправка уведомлений в Telegram, ack/ретрай/DLQ).

## Пользователь получает
Как **покупатель**, я могу **оформить заказ и получить уведомление**, чтобы **быть уверенным, что заказ принят**, — при этом заказ и уведомление атомарны (ADR-004).

## Готово когда
- [ ] Given оформление, when вызывается `POST /orders`, then состав берётся из корзины, остаток списывается под `SELECT ... FOR UPDATE`, заказ и `order_items` пишутся снимками — всё в одной транзакции
- [ ] Given транзакция, when она коммитится, then событие лежит в `outbox`; из HTTP-хендлера в брокер ничего не публикуется
- [ ] Given релей, when брокер доступен, then он публикует неопубликованные события и помечает их published
- [ ] Given потребитель, when доставка повторяется, then второе уведомление не отправляется (идемпотентность)
- [ ] Given исчерпание попыток, when задача падает, then она уходит в DLQ и по ней срабатывает алерт

## Как делаем
- Одна транзакция в `orders`; расчёт доставки — через `delivery` (методы + quote).
- Релей: воркер опрашивает `outbox` (частичный индекс), публикует, помечает.
- Consumer: ack/nack с задержкой, DLQ; идемпотентность по ключу события.

## Файлы
- `backend/internal/orders/**`, `backend/internal/delivery/**`, `backend/internal/outbox/**`, `backend/internal/notifications/**`
- `backend/cmd/worker/main.go` — запуск релея и потребителей
