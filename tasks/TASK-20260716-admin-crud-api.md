# TASK-20260716-admin-crud-api: Админский модуль (CRUD контента, каталога, заказов, настроек)

**Type:** feature  
**Size:** L  
**Status:** todo
**GitHub Issue:** #10 — https://github.com/prog-time/tg-shop/issues/10

## Что делаем
Реализовать модуль `admin`: CRUD товаров, категорий, справочников и их значений, реестра полей, новостей, способов доставки и оплаты; ведение заказов (список, смена статуса по разрешённым переходам); настройки, роли и учётки персонала. Админка не имеет привилегированного доступа к БД — вызывает те же доменные сервисы, что и витрина, поэтому инварианты соблюдаются одинаково.

## Пользователь получает
Как **менеджер/администратор**, я могу **вести каталог, справочники, новости и заказы**, чтобы **управлять магазином без разработчика и деплоя** (ADR-002).

## Готово когда
- [ ] Given CRUD каталога, when создаётся/меняется товар, then `catalog` валидирует `attributes` по `field_definitions` и пересобирает `product_facets`
- [ ] Given справочники/поля, when значение или поле используется, then оно архивируется/деприкейтится, а не удаляется
- [ ] Given заказ, when меняется статус, then допускаются только разрешённые переходы (одинаково для витрины и админки)
- [ ] Given RBAC, when роль не имеет права, then `403` (admin / order_manager / content_manager)
- [ ] Given запись каталога, when она произошла, then кэш витрины инвалидируется

## Как делаем
- `adminmod` вызывает доменные сервисы `catalog`/`content`/`orders`/`delivery`/`payment`, не пишет в БД в обход них.
- Смена статуса заказа — через доменный сервис с проверкой переходов.
- Настройки/роли/учётки — CRUD с RBAC.
