# ADR-005. Набор Go-библиотек и контракт-first через oapi-codegen

**Статус:** Accepted

## Контекст

Стек проекта фиксировал язык (Go, один модуль, три entrypoint-а), но не библиотеки — `docs/architecture.md` и `CLAUDE.md` прямо предписывали обсудить их до кода. Перед стартом реализации (walking skeleton) выбор нужно закрепить: он формирует скелет, который наследует весь бэкенд. Дополнительная развилка: у нас уже есть контракт `docs/api/openapi.yaml` — генерировать сервер из него или писать хендлеры руками.

Ограничение из `CLAUDE.md`: в ядре только свободные лицензии; перед добавлением зависимости проверяется лицензия.

## Решение

Минимальный набор проверенных библиотек, всё — под пермиссивными лицензиями (MIT / BSD / Apache-2.0), совместимыми с требованием свободного ядра:

| Назначение | Библиотека | Лицензия |
|---|---|---|
| HTTP-роутер + middleware | `go-chi/chi/v5` | MIT |
| Драйвер и пул PostgreSQL | `jackc/pgx/v5` (+ `pgxpool`) | MIT |
| Миграции схемы | `pressly/goose/v3` | MIT |
| Redis-клиент | `redis/go-redis/v9` | BSD-2-Clause |
| AMQP / RabbitMQ | `rabbitmq/amqp091-go` | BSD-2-Clause |
| JWT (админка) | `golang-jwt/jwt/v5` | MIT |
| Метрики Prometheus | `prometheus/client_golang` | Apache-2.0 |
| Кодогенерация из OpenAPI | `oapi-codegen/oapi-codegen/v2` | Apache-2.0 |

Логирование — стандартный `log/slog` (JSON в stdout). Конфигурация — stdlib (`os.Getenv` + тонкий загрузчик), без отдельной библиотеки, чтобы не плодить зависимости.

**Контракт-first.** `docs/api/openapi.yaml` — источник правды для HTTP-слоя. Из него `oapi-codegen` генерирует модели и серверные интерфейсы (`StrictServerInterface`); доменные модули реализуют эти интерфейсы. Спецификация правится раньше кода; сгенерированный код в репозиторий коммитится (воспроизводимость сборки без сети).

## Обоснование

- **chi**, а не фреймворк (Echo/Gin/Fiber): совместим со стандартным `net/http`, минимальная магия, тонкий слой middleware — под модульный монолит (ADR-003) это важнее «батареек».
- **pgx**, а не `database/sql`+ORM: прямой доступ к PostgreSQL, лучшая производительность и типы; ORM противоречил бы принципу «владелец схемы и SQL — Go API». Никакого скрытого генератора запросов.
- **goose**: простые SQL-миграции, встраиваются в бинарь через `embed`; api применяет их на старте — он единственный владелец схемы.
- **oapi-codegen** закрывает риск расхождения кода и контракта: типы и сигнатуры хендлеров генерируются, ручная часть — только доменная логика. Kafka-подобные тяжёлые тулчейны не нужны.
- Всё перечисленное — де-факто стандарт экосистемы Go, активно поддерживается, без copyleft-обязательств.

## Следствия

- Версия Go и точные версии библиотек живут в `backend/go.mod`; этот ADR фиксирует *выбор*, а не версии.
- Добавление новой зависимости — с проверкой лицензии и, если она меняет архитектуру (новый слой, брокер, ORM), с новым ADR.
- Смена HTTP-фреймворка или драйвера БД задним числом — новый ADR со статусом `Superseded`.
- Генерация: `go generate ./...` в `backend/` пересобирает код из `docs/api/openapi.yaml`.

## Аддендум (issue #4): нормализация nullable-идиом OpenAPI 3.1 перед кодогенерацией

**Контекст.** `docs/api/openapi.yaml` написан на OpenAPI 3.1 и повсеместно использует 3.1-идиомы для nullable-полей: `type: [X, "null"]` и `oneOf`/`anyOf: [<schema>, {type: null}]`. `oapi-codegen/oapi-codegen/v2` (зафиксирован этим ADR) пока не понимает 3.1 type-объединения — известное ограничение, [oapi-codegen/oapi-codegen#373](https://github.com/oapi-codegen/oapi-codegen/issues/373) — и падает с ошибкой генерации на первом же таком поле. Контракт при этом остаётся источником правды и не переписывается под ограничения генератора.

**Решение.** Перед вызовом `oapi-codegen` контракт прогоняется через `backend/internal/openapi/specprep` — небольшую hand-written Go-программу, которую `go generate` запускает первым шагом (см. `//go:generate` в `backend/internal/openapi/generate.go`). Она читает `docs/api/openapi.yaml`, переписывает **ровно два** распознаваемых идиома в 3.0-совместимую форму и пишет результат во временный, гитигнорируемый файл — `docs/api/openapi.yaml` при этом не модифицируется:

| Идиома OpenAPI 3.1 | Переписывается в |
|---|---|
| `type: [X, "null"]` | `type: X` + `nullable: true` |
| `oneOf: [<schema>, {type: null}]` (ровно два элемента, один — голый `{type: null}`) | `allOf: [<schema>]` + `nullable: true` |
| `anyOf: [<schema>, {type: null}]` (та же форма) | `allOf: [<schema>]` + `nullable: true` |

Любая другая форма — трёхэлементное объединение, `oneOf` из одного голого `{type: null}`, nullable-enum, обёрнутый иначе, и т. п. — **не переписывается и не гадается**, а оставляется как есть.

**Политика для нераспознанных форм.** Такое решение опасно тихой деградацией: если контракт в будущем добавит новую nullable-идиому, `specprep` молча пропустит её как литеральный YAML, а `oapi-codegen` либо упадёт с непонятной ошибкой глубоко в резолвере схем, либо — хуже — сгенерирует код, не соответствующий контракту, без единой ошибки. Поэтому в `backend/internal/openapi/specprep/main_test.go` есть guard-тест (`TestNormalize_RealContractHasNoUnhandledNullableShapes`): он гоняет `normalize()` против настоящего `docs/api/openapi.yaml` и падает, если после нормализации остался хоть один необработанный `type`-массив или `oneOf`/`anyOf` с необработанной null-веткой. Любое будущее изменение контракта, вводящее новую nullable-идиому, ловится этим тестом до кодогенерации, а не после.

**Побочный эффект.** Сгенерированный `GetSwagger()`/embedded spec в `internal/openapi/openapi.gen.go` отдаёт нормализованную (специреповскую) копию спецификации, а не побайтовый оригинал 3.1-контракта. Сегодня это ничем не грозит (никто эту функцию не вызывает), но важно помнить, если позже кто-то подключит валидацию запросов по схеме или отдачу `/openapi.json` из этого кода — это будет не тот же файл, что лежит в `docs/api/openapi.yaml`.

**Следствие.** Если `oapi-codegen` в будущей версии добавит поддержку OpenAPI 3.1 type-объединений (закрытие #373), `specprep` и этот аддендум становятся историческими — шаг убирается, а `oapi-codegen` запускается напрямую на `docs/api/openapi.yaml`.
