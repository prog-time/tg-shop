# ADR-002. Гибкость каталога: реестр полей, справочники, JSONB

**Статус:** Accepted

## Контекст

Характеристики товара зависят от категории: у обуви размер, у мебели материал и форма. Менеджеры заводят справочники вручную (Тип товара, Материал, Форма) и постоянно их расширяют, а затем заполняют поля товара значениями из справочников. Покупатель фильтрует каталог по этим же характеристикам.

Требование «менеджер добавляет поле или справочник без участия разработчика и без деплоя» исключает подход «каждая характеристика — колонка в таблице».

## Варианты

**1. Колонка на характеристику.** Каждое новое поле — миграция и деплой.

**2. Классический EAV** (таблица «сущность — атрибут — значение»).

**3. Реестр полей + JSONB + справочники.**

## Решение

Выбран вариант 3.

Схема:

- `dictionaries` и `dictionary_values` — справочники и их значения, заводятся менеджером в рантайме.
- `field_definitions` — реестр полей: к какой категории относится, тип, обязательность, валидация, порядок в форме, показывать ли колонкой в списке. Среди типов есть «значение из справочника» и «несколько значений из справочника».
- `products.attributes JSONB` — сами значения характеристик товара.
- `product_facets` — служебная таблица связей «товар ↔ значение справочника», пересобирается при каждой записи товара.

Админка **рендерит форму из реестра**: на каждый тип поля — один Vue-компонент. Добавили поле в категорию — оно появилось в форме, без пересборки фронтенда. Go валидирует `attributes` по реестру при записи.

## Обоснование

Вариант 1 нарушает главное требование: каждая новая характеристика превращается в поход к разработчику.

Вариант 2 даёт гибкость, но делает любой запрос каталога набором джойнов и убивает читаемость. Классический EAV — известный источник неподдерживаемых проектов.

Вариант 3 разделяет ответственность правильно: JSONB отвечает за отображение, `product_facets` — за поиск. Фасетные фильтры на витрине работают обычным индексом по обычной таблице, а не разбором JSON на каждый запрос.

Ключевое: это **не самописная CMS**. Динамические поля живут ровно в одном месте — характеристики товара, — там, где изменчивость реальна. Всё остальное (новости, доставка, оплата, заказы) — обычные таблицы с обычными формами.

## Правила, без которых схема сгниёт

- **В `attributes` хранится id значения справочника, а не текст.** Иначе переименование «Хлопок» → «Хлопок 100%» породит зоопарк из «хлопок», «Хлопок» и «хлопок ».
- **Значение справочника не удаляется, а архивируется**, если используется. Архивное не предлагается в новых товарах, но корректно отображается в старых.
- **Поле не удаляется, а помечается устаревшим.** Смена типа существующего поля запрещена: создаётся новое, данные переносятся.
- **Поле, по которому фильтруют или сортируют, — это колонка или запись в `product_facets`**, а не поиск по JSONB.

## Следствия

- Менеджер расширяет каталог без деплоя — то, чего не даёт даже Strapi (его конструктор типов не работает в production).
- Ценой является рендер формы по схеме: ~8–9 компонентов полей в админке.
- Валидация значений — ответственность Go. «В JSONB что угодно» недопустимо.
